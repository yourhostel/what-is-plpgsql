# what-is-plpgsql [![PL/pgSQL Documentation](https://img.shields.io/badge/PostgreSQL-PL%2FpgSQL_Documentation-336791?logo=postgresql&logoColor=white)](https://www.postgresql.org/docs/current/plpgsql.html)

**Огляд PL/pgSQL, процедурної мови для PostgreSQL. 
(Overview of PL/pgSQL, the procedural language for PostgreSQL)**

`PL pgSQL` це вбудована процедурна мова `PostgreSQL` для написання серверної логіки всередині самої СУБД. 
Повна назва, **Procedural Language PostgreSQL Structured Query Language**. Її призначення, "дати" `PostgreSQL` 
окрім декларативного `SQL` для вибірки та зміни даних, ще й можливість описувати послідовність дій, 
умови, цикли, змінні, обробку помилок, курсори, тригери та функції.
`PL pgSQL` це не окремий зовнішній плагін, не окрема програма, не замінює `SQL`. Це вбудована 
процедурна надбудова над `SQL`, інтегрована в `PostgreSQL` як мова серверного програмування. Вона працює 
всередині процесу бази даних і виконується на сервері поруч із даними.

## Використання

Це клас завдань, де чистого SQL недостатньо або він стає громіздким:

1. потрібно виконати кілька послідовних дій
2. потрібно зберігати проміжні значення в змінних
3. потрібно робити перевірки if then else
4. потрібно проходити набір рядків циклом
5. потрібно перехоплювати помилки
6. потрібно реалізувати тригерну логіку до або після змін у таблиці
7. потрібно інкапсулювати бізнес правило на стороні БД
8. потрібно повертати складний набір результатів з логікою

## Архітектура

У PostgreSQL є ядро виконання `SQL`, планувальник, оптимізатор, механізми транзакцій, блокувань, індексів, 
зберігання таблиць. Поверх цього є механізм серверних мов. Одна з них, `PL pgSQL`. Тобто це серверний механізм у межах `PostgreSQL`.
Коли створюємо функцію на `PL pgSQL`, `PostgreSQL`:

1. зберігає її текст у системному каталозі
2. знає, що її треба виконувати інтерпретатором `PL pgSQL`
3. при виклику функції створює контекст виконання
4. компілює або готує внутрішнє представлення блоків
5. виконує `SQL` оператори всередині функції з доступом до змінних, умов, циклів та винятків

## Найтиповіші сценарії

1. Функції, які повертають скалярне значення,
   - Наприклад, порахувати коментарі, визначити статус, обчислити підсумок.
2. Функції, які повертають таблицю або набір рядків,
   - Наприклад, складний серверний пошук.
3. Тригери,
   - Наприклад, автоматично оновити updated_at, записати аудит, перевірити інваріанти.
4. Серверні процедури,
   - Наприклад, виконати пакетну обробку.
5. Логіка перевірок і централізованих правил,
   - Наприклад, заборонити негативний баланс або дублікати в складніших випадках.
6. Обробка помилок
   - Наприклад, перехопити виняток, зробити fallback, повернути код помилки.

## Краще не використовувати

`PL pgSQL` не слід перетворювати на повноцінну заміну прикладного сервісу.

Краще залишати зовні:

1. складна бізнес логіка, тісно пов’язана з `UI`
2. інтеграція з `HTTP API`
3. робота з файлами, чергами, зовнішніми сервісами
4. великий доменний застосунок з багатим шаром об’єктної логіки
5. важке `CPU` обчислення, яке простіше масштабувати поза БД

## Базова форма функції

```sql
CREATE OR REPLACE FUNCTION function_name(param_name param_type)
RETURNS return_type AS $$
DECLARE
    variable_name variable_type;
BEGIN
    -- оператори
    RETURN ...;
END;
$$ LANGUAGE plpgsql;
```

> - `CREATE OR REPLACE FUNCTION` створює або перевизначає функцію
> - `RETURNS` задає тип результату
> - `AS $$ ... $$` містить тіло функції
> - `DECLARE` секція змінних
> - `BEGIN ... END` секція виконання
> - `LANGUAGE plpgsql` каже, якою мовою написане тіло

## Опис функцій і процедур

| Конструкція        | Категорія  | Призначення         | Стислий опис                   | Приклад                               |
|--------------------|------------|---------------------|--------------------------------|---------------------------------------|

| `CREATE PROCEDURE` | DDL        | Створення процедури | Реєструє процедуру             | `CREATE PROCEDURE p() ...`            |
| `RETURNS`          | Сигнатура  | Тип результату      | Визначає тип повернення        | `RETURNS bigint`                      |
| `RETURNS TABLE`    | Сигнатура  | Табличний результат | Описує структуру набору рядків | `RETURNS TABLE(id int)`               |
| `OUT` параметри    | Сигнатура  | Вихідні параметри   | Повернення через параметри     | `OUT result int`                      |
| `LANGUAGE plpgsql` | Метадані   | Вказівка мови       | Визначає мову виконання        | `LANGUAGE plpgsql`                    |
| `AS $$ ... $$`     | Оформлення | Тіло функції        | Містить код PL pgSQL           | `AS $$ BEGIN ... END; $$`             |

## Типові шаблони використання

### Повернути одне значення

Функція як серверний обчислювач.

```sql
CREATE OR REPLACE FUNCTION public.count_comments(postid bigint)
RETURNS bigint AS $$
DECLARE
    v_comment_count bigint;
BEGIN
    SELECT COUNT(*)
    INTO v_comment_count
    FROM post_comment
    WHERE post_id = postid;

    RETURN v_comment_count;
END;
$$ LANGUAGE plpgsql;
```

###  Перевірити умову і згенерувати помилку

База сама захищає інваріант.

```sql
CREATE OR REPLACE FUNCTION public.ensure_positive_score(p_score integer)
RETURNS void AS $$
BEGIN
    IF p_score < 0 THEN
        RAISE EXCEPTION 'Score не може бути від’ємним';
    END IF;
END;
$$ LANGUAGE plpgsql;
```

### Повернути набір рядків

```sql
CREATE OR REPLACE FUNCTION public.get_post_comments(postid bigint)
RETURNS TABLE(id bigint, content text) AS $$
BEGIN
    RETURN QUERY
    SELECT pc.id, pc.content
    FROM post_comment pc
    WHERE pc.post_id = postid;
END;
$$ LANGUAGE plpgsql;
```

### Тригер

функція не викликається напряму користувачем, а виконується механізмом тригера.

```sql
CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS trigger AS $$
BEGIN
    NEW.updated_at := CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## Механізм створення, зберігання і виклику

У PostgreSQL функції і процедури створюються через `SQL` і стають частиною схеми бази даних.

### Створення

Функція або процедура створюється через `DDL` команду:
```sql
CREATE OR REPLACE FUNCTION function_name(...)
    RETURNS ...
    AS $$
BEGIN
    ...
END;
$$ LANGUAGE plpgsql;
```

- Після виконання цієї команди:

  1. функція реєструється в системних каталогах
  2. стає доступною як об’єкт БД
  3. може викликатися з будь якого клієнта

> Функція або процедура  не є файлами у файловій системі, а є частиною метаданих БД.

- Зберігання:

1. зберігаються всередині бази даних
2. записуються в системні таблиці `PostgreSQL`
3. переживають перезапуск сервера
4. зникають тільки при явному видаленні або втраті даних

### Виклик

#### Функція викликається через `SELECT`:

```sql
SELECT function_name(...);
```

#### Або як джерело даних:

```sql
SELECT * FROM function_name(...);
```

#### Процедура викликається через `CALL`:

```sql
CALL procedure_name(...);
```

## Порівняння функцій і процедур

| Характеристика          | Функція                                                    | Процедура                                 |
|-------------------------|------------------------------------------------------------|-------------------------------------------|
| Призначення             | Обчислення і повернення значення                           | Виконання дій                             |
| Виклик                  | `SELECT function(...)`, `SELECT * FROM function_name(...)` | `CALL procedure(...)`                     |
| Повернення значення     | Обов’язкове                                                | Не обов’язкове                            |
| Тип результату          | Скаляр або набір рядків                                    | Немає або через OUT                       |
| Використання в SQL      | Можна в `SELECT`, `WHERE`, `JOIN`                          | Не можна                                  |
| Робота з транзакціями   | Не керує транзакціями                                      | Може використовувати `COMMIT`, `ROLLBACK` |
| Типові сценарії         | Агрегації, обчислення, вибірки                             | Пакетні операції, зміни стану             |
| Використання в тригерах | Так                                                        | Ні                                        |
| Інтеграція з ORM        | Як частина запиту                                          | Як окремий виклик                         |
| Поведінка               | Функціональна                                              | Імперативна                               |

## Приклад використання у ORM (Hibernate, JPA)

Приклад описано у статі `Vlad Mihalcea`

[How to map a JPA entity to a View or SQL query using Hibernate](https://vladmihalcea.com/map-jpa-entity-to-view-or-sql-query-with-hibernate/)

У практиці `ORM` (наприклад, `Hibernate`) не завжди достатньо автоматично згенерованого `SQL`. У таких випадках:

1. складна логіка реалізується на стороні `PostgreSQL` (`SQL` або функції)
2. результат цієї логіки мапиться на `entity`

`Entity` можна прив’язати не до таблиці, а до результату запиту або функції через `@Subselect`:

```sql
SELECT 
    routines.routine_name,
    parameters.data_type
FROM information_schema.routines
...
```

`PostgreSQL` виконує `SQL` або функцію і повертає результат. `ORM` отримує цей результат і мапить його на об’єкти.
